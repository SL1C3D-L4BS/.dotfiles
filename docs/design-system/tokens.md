## Design Tokens — SL1C3D-L4BS (Phase 4)

This document references:

- Phase 0 inventory and decisions: `docs/inventory/phase0-inventory.md`, `docs/inventory/phase0-decisions.md`
- Phase 0 contradictions: `docs/inventory/phase0-contradictions.md`

Phase 4 implements the **three-layer theme architecture** with a single authoritative generator. No runtime artifact is canonical.

---

### 1. Three-Layer Architecture

- **Layer A — Canonical repo theme tokens**  
  The only source of truth is in the repository: **`theme.tokens.toml`** at repo root. It defines colors, spacing, radius, motion durations (120 / 180 / 260 ms), typography, and semantic UI roles. No other file may define theme constants independently.

- **Layer B — Generated runtime theme JSON**  
  **`~/.config/sl1c3d/theme.json`** is generated from Layer A by the theme generator. It is disposable, never canonical, and excluded from git. Runtime tools may read it; they may not write back into the canonical repo tokens.

- **Layer C — Consumer-specific outputs**  
  All consumer artifacts are generated from Layer A via the single generator pipeline: Quickshell `BrandTheme.qml`, AGS `sl1c3d-tokens.scss`, Hyprland `colors.conf`, Ghostty theme `sl1c3d-l4bs`. No consumer component may define independent theme constants.

---

### 2. Authoritative Generator Pipeline

- **Single entry point:** **`sl1c3d theme apply`** is the only public way to run the theme pipeline.
- **Generator location:** `dot_config/SL1C3D-L4BS/theme/generate.sh` (wrapper) and `dot_config/SL1C3D-L4BS/theme/generate.py` (implementation). The generator may call matugen internally if required; matugen is not a second authority. No independent theme scripts exist outside this pipeline.

---

### 3. Theme File Contract

| Item | Path / rule |
|------|------------------|
| **Canonical token source** | `theme.tokens.toml` at repo root (chezmoi source root). |
| **Generator script** | `dot_config/SL1C3D-L4BS/theme/generate.sh` (invokes `generate.py`). |
| **Generated runtime JSON** | `~/.config/sl1c3d/theme.json`. |
| **Generated consumer outputs** | `dot_config/quickshell/bar/BrandTheme.qml`, `dot_config/ags/sl1c3d-tokens.scss`, `dot_config/hypr/colors.conf`, `dot_config/ghostty/themes/sl1c3d-l4bs`. |
| **Committed to git** | `theme.tokens.toml`, `dot_config/.../theme/generate.sh`, `generate.py`, and all generated consumer files above (they are regenerated from tokens and committed so the repo is self-consistent). |
| **Ignored** | `dot_config/sl1c3d/theme.json` if ever present under repo; runtime `~/.config/sl1c3d/theme.json` is never in the repo. |
| **Regenerated by `sl1c3d theme apply`** | Runtime JSON plus all four consumer files listed above. |

---

### 4. Motion Standardization

All consumer outputs use:

- **fast:** 120 ms  
- **normal:** 180 ms  
- **slow:** 260 ms  

These values are defined only in `theme.tokens.toml` (section `[motion]`) and propagated by the generator. No conflicting motion constants remain in BrandTheme.qml, sl1c3d-tokens.scss, or other consumer files after generation.

---

### 5. Token Categories in theme.tokens.toml

- **colors:** bg, surface, border, fg, accent, logo, error, success, warn, glass variants.  
- **spacing:** xs, sm, md, lg, xl.  
- **radius:** pill, modal, card.  
- **motion:** fast_ms, normal_ms, slow_ms (120, 180, 260).  
- **typography:** font_family, font_size_base.  
- **semantic:** UI roles referencing color keys.

---

### 6. CI Compatibility

The theme pipeline is compatible with Phase 2 CI render-sanity: all Phase 0 render targets (Hyprland config, Quickshell QML fragments, AGS theme surface, Starship config, runtime theme JSON path) still pass. Generated consumer files are committed and contain no unresolved template markers.

---

### 7. Rollback

If the generator or token model causes regressions: revert changes to `theme.tokens.toml` and the generator scripts, re-run `sl1c3d theme apply`, and commit the regenerated consumer files. Legacy hard-coded theme assets are no longer present; the single source of truth is `theme.tokens.toml`.
